# Описание сервисов

### 1. UI (Frontend)
**Назначение:** Предоставляет интерфейс для операторов, для учета аппаратных модулей безопасности.

**Функциональность:**
- Форма создания HSM (поля: `name`, `comment`).
- Форма редактирования HSM (поля: `name`, `comment`).
- Взаимодействует **только** с `service1`. О работе HSM ничего не знает.

**Технологии:**
- статический HTML + JS.

## 2. service1 (Core Backend)
**Назначение:** Основной API-шлюз для UI, выполняет валидацию и управляет жизненным циклом HSM.

**Основные эндпоинты (REST):**

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/v1/hsm` | Принимает запрос на создание HSM. Валидирует входные данные. Сохраняет запись в БД |
| GET | `/api/v1/hsm` | Возвращает список всех HSM с пагинацией. |
| GET | `/api/v1/hsm/{id}` | Возвращает информацию о конкретном HSM. |
| PUT | `/api/v1/hsm/{id}` | Редактирует значение HSM |

**C HSM и service2 не взаимодействует**

**База данных:**
- Таблица `hsm`: `id`, `name`, `comment`, `created_at`, `updated_at`.

**Валидация:**
- Имя: не пустое, не более 100 символов, только буквы/цифры/подчеркивание.

---

## 3. service2 (HSM Worker)
**Назначение:** Изолированный микросервис, который имеет прямые интеграции с HSM. Не имеет внешнего API для UI, доступен только для  вызовов от внешних сервисов (не `service1`).

**Функциональность:**
- Генерация ключевых пар (для RSA) или симметричных ключей (для AES) с использованием симулированного HSM.
- Возврат сгенерированного ключа (или его публичной части) и уникального идентификатора в HSM.
- Поддержка операций:
  - `POST /internal/hsm/generate` – принимает `algorithm`, `keyLength`, `keyId` (внешний ID). Возвращает `hsmReference` и публичный материал (для RSA).
  - `GET /internal/hsm/status/{hsmReference}` – проверка статуса ключа в HSM.
  - `POST /internal/hsm/revoke/{hsmReference}` – отзыв ключа в HSM.
