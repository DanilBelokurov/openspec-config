# Тестовый проект: Система управления ключами HSM

## Общая концепция

Проект представляет собой микросервисную архитектуру для управления криптографическими ключами, генерируемыми аппаратным модулем безопасности (HSM). Система состоит из трёх компонентов:

- **UI (Frontend)** – веб-интерфейс для операторов.
- **service1 (Backend API)** – основной бэкенд, обрабатывающий запросы от UI, выполняющий бизнес-логику, валидацию и сохранение данных в БД.
- **service2 (HSM Worker)** – изолированный бэкенд-сервис, напрямую взаимодействующий с HSM (симулятором или реальным устройством) для генерации и выпуска ключей. **Не имеет доступа к UI и работает только по внутренним вызовам от service1.**

Все бэкенд-сервисы написаны на **Kotlin + Spring Boot 3.4 (Spring Boot 4 условно)**, используют **Gradle/Kotlin DSL**, **PostgreSQL** и **REST/gRPC** для общения (в тестовом проекте используется REST).

## Описание сервисов

### 1. UI (Frontend)
**Назначение:** Предоставляет интерфейс для операторов, чтобы инициировать выпуск ключей и просматривать их статус.

**Функциональность:**
- Форма создания заявки на выпуск ключа (поля: `name`, `algorithm` (RSA/AES), `keyLength`, `comment`).
- Таблица со списком всех созданных ключей и их статусами (`PENDING`, `PROCESSING`, `ACTIVE`, `REVOKED`, `ERROR`).
- Кнопка обновления статуса (ручной опрос).
- Взаимодействует **только** с `service1`. О работе HSM ничего не знает.

**Технологии:** (на усмотрение агента, но в проекте предполагается статический HTML + JS для простоты, либо отдельный контейнер с Nginx).

---

### 2. service1 (Core Backend)
**Назначение:** Основной API-шлюз для UI, управляет жизненным циклом ключей, выполняет валидацию и оркестрацию вызовов к `service2`.

**Основные эндпоинты (REST):**

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/v1/keys` | Принимает запрос на создание ключа. Валидирует входные данные (алгоритм, длина, имя). Сохраняет запись в БД со статусом `PENDING`. Асинхронно отправляет задачу в `service2`. |
| GET | `/api/v1/keys` | Возвращает список всех ключей с пагинацией. |
| GET | `/api/v1/keys/{id}` | Возвращает информацию о конкретном ключе. |
| GET | `/api/v1/keys/{id}/status` | Проверяет статус ключа (может опросить `service2` или взять из БД). |
| POST | `/api/v1/keys/{id}/revoke` | Отзывает ключ (меняет статус на `REVOKED`, если это позволяет состояние). |

**Взаимодействие с HSM (service2):**
- При создании ключа `service1` отправляет асинхронный запрос (через `@Async` или `WebClient` с таймаутом) в `service2` на генерацию.
- Не блокирует ответ для UI – сразу возвращает `202 Accepted` с `id` заявки.
- Периодически (или по запросу) обновляет статус ключа в БД, получая ответ от `service2`.

**База данных:**
- Таблица `crypto_keys`: `id`, `name`, `algorithm`, `key_length`, `comment`, `status` (enum), `public_key` (text), `created_at`, `updated_at`, `hsm_reference` (идентификатор ключа в HSM).

**Валидация:**
- Имя: не пустое, не более 100 символов, только буквы/цифры/подчеркивание.
- Алгоритм: только `RSA` или `AES`.
- Длина: для RSA — 1024, 2048, 4096; для AES — 128, 192, 256.

---

### 3. service2 (HSM Worker)
**Назначение:** Изолированный микросервис, который имеет прямые интеграции с HSM (в тестовом проекте — симулятор). Не имеет внешнего API для UI, доступен только для внутренних вызовов от `service1`.

**Функциональность:**
- Генерация ключевых пар (для RSA) или симметричных ключей (для AES) с использованием симулированного HSM.
- Возврат сгенерированного ключа (или его публичной части) и уникального идентификатора в HSM.
- Поддержка операций:
  - `POST /internal/hsm/generate` – принимает `algorithm`, `keyLength`, `keyId` (внешний ID). Возвращает `hsmReference` и публичный материал (для RSA).
  - `GET /internal/hsm/status/{hsmReference}` – проверка статуса ключа в HSM.
  - `POST /internal/hsm/revoke/{hsmReference}` – отзыв ключа в HSM.

**Особенности:**
- Имитация задержки генерации (от 2 до 5 секунд) – чтобы показать асинхронность.
- Все операции логируются в отдельный файл `hsm_audit.log`.
- Не имеет собственной базы данных, хранит только кеш состояний в памяти (или Redis для демонстрации).

## Инфраструктура и запуск

### Стек
- **Java 21** + **Kotlin** (1.9+)
- **Spring Boot 3.4** (условно версия 4, но используется стабильная 3.x)
- **PostgreSQL 16** (общая БД для service1)
- **Docker / Docker Compose** для сборки окружения (включая симулятор HSM).
- **Gradle** (Kotlin DSL) для сборки.

### Взаимодействие между сервисами
- Синхронное REST (HTTP/2) с использованием `WebClient` с таймаутами (5 сек).
- В реальном проекте можно заменить на gRPC, но в тестовом используется REST для простоты.
